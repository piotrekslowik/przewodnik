🔤Nagłówki
Markdown obsługuje 6 poziomów nagłówków odpowiadających HTML <h1>–<h6>. Prefiks # — liczba znaków określa poziom. Zawsze zostaw spację po znakach #.
Markdown 
# Nagłówek H1
## Nagłówek H2
### Nagłówek H3
#### Nagłówek H4
##### Nagłówek H5
###### Nagłówek H6
Podgląd

Nagłówek H1

Nagłówek H2

Nagłówek H3

Nagłówek H4

Nagłówek H5
Nagłówek H6
Alternatywna składnia (Setext) dla H1 i H2
Markdown
Tytuł dokumentu
===============

Podtytuł
--------
Podgląd

Tytuł dokumentu

Podtytuł

Uwaga: H1 powinien wystąpić tylko raz w dokumencie — jest tytułem strony. Kolejne sekcje zaczynaj od H2.
✏️Formatowanie tekstu
Podstawowe style: pogrubienie, kursywa, przekreślenie, podkreślenie (HTML), indeksy oraz kod inline.
Markdown 
**pogrubiony**
*kursywa*
***pogrubiona kursywa***
~~przekreślony~~
`kod inline`
<u>podkreślony</u>
H~2~O  (GFM / Obsidian)
x^2^   (GFM / Obsidian)
Podgląd

pogrubiony

kursywa

pogrubiona kursywa

przekreślony

kod inline

podkreślony

H2O

x2

Akapity i łamanie linii
Markdown
Pierwszy akapit tekstu.

Drugi akapit — pusta linia tworzy akapit.

Pierwsza linia··
Druga linia (2 spacje = <br>)
Podgląd

Pierwszy akapit tekstu.

Drugi akapit — pusta linia tworzy akapit.

Pierwsza linia
Druga linia (2 spacje = <br>)

Tip: Możesz też używać _kursywa_ i __pogrubienie__ z podkreślnikami — ale gwiazdki są bardziej universalne i rzadziej powodują problemy w środku słów.
📋Listy
Nieuporządkowane (punktorowe), uporządkowane (numerowane), zagnieżdżone i mieszane. Wcięcie 2 lub 4 spacje tworzy poziom zagnieżdżenia.
Lista nieuporządkowana
Markdown
- Element A
- Element B
  - Zagnieżdżony 1
  - Zagnieżdżony 2
    - Głębszy poziom
- Element C
Podgląd
  • Element A
  • Element B
    • Zagnieżdżony 1
    • Zagnieżdżony 2
      • Głębszy poziom
  • Element C
Lista uporządkowana
Markdown
1. Krok pierwszy
2. Krok drugi
   1. Podkrok 2a
   2. Podkrok 2b
3. Krok trzeci
Podgląd
  1. Krok pierwszy
  2. Krok drugi
    1. Podkrok 2a
    2. Podkrok 2b
  3. Krok trzeci
Uwaga: Liczby w liście numerowanej nie muszą być kolejne — Markdown sam je ponumeruje. Możesz pisać 1. 1. 1. i efekt będzie taki sam.
🔗Linki i obrazy
Linki inline, referencyjne i automatyczne. Obrazy używają tej samej składni co linki, poprzedzone !.
Markdown 
[tekst linku](https://example.com)
[link z tooltipem](https://example.com "Opis")
https://example.com  (autolink)


[referencyjny][ref1]

[ref1]: https://example.com "Opis referencji"
Podgląd

tekst linku

link z tooltipem

https://example.com

referencyjny

Obrazy
Markdown
![tekst alternatywny](obraz.png)
![logo](logo.png "Tooltip obrazka")


[![alt](thumb.png)](https://example.com)


<img src="img.png" width="300" alt="opis">
Podgląd

🖼️ [obraz.png] — tekst alternatywny

Obraz jako link: kliknięcie obrazka przenosi na URL.

Tip: Tekst alternatywny (alt) jest obowiązkowy dla dostępności (a11y) i SEO. Zawsze opisuj co jest na obrazku.
💬Cytaty (Blockquote)
Prefiks > tworzy blok cytatu. Zagnieżdżone cytaty — wielokrotny prefiks. Wewnątrz cytatu można używać pełnego Markdown.
Markdown
> To jest cytat.
> Może obejmować wiele linii.
>
> Drugi akapit cytatu.
>
> > Zagnieżdżony cytat
> > drugiego poziomu.
Podgląd

To jest cytat. Może obejmować wiele linii.

Drugi akapit cytatu.

Zagnieżdżony cytat drugiego poziomu.

💻Kod
Kod inline (backtick) i bloki kodu (triple backtick z nazwą języka). Bloki kodu obsługują syntax highlighting na GitHub, GitLab i w większości rendererów.
Kod inline
Markdown
Użyj `npm install` aby zainstalować.
Zmienna `const x = 42` jest stała.
Podgląd

Użyj npm install aby zainstalować.

Zmienna const x = 42 jest stała.

Blok kodu z podświetlaniem składni
Markdown
```python
def hello(name: str) -> str:
    return f"Hello, {name}!"

print(hello("World"))
```
Podgląd
def hello(name: str) -> str:
    return f"Hello, {name}!"

print(hello("World"))
Popularne identyfikatory języków
Języki
javascript / js    typescript / ts    python / py
bash / sh          sql                json
html               css                yaml / yml
rust               go                 java
c / cpp            csharp             php
markdown / md      dockerfile         diff
Blok diff (zmiany)
Markdown
```diff
- stara linia (usunięta)
+ nowa linia (dodana)
  linia bez zmian
```
Podgląd
- stara linia (usunięta)
+ nowa linia (dodana)
  linia bez zmian
Uwaga: Blok kodu można też tworzyć wcięciem o 4 spacje (stara składnia), ale triple-backtick jest powszechniej stosowany i obsługuje określenie języka.
📊Tabele
Tabele GFM (GitHub Flavored Markdown). Wyrównanie kolumn kontroluje się dwukropkiem w linii separatora.
Markdown 
| Kolumna L | Kolumna C | Kolumna P |
| :-------- | :-------: | --------: |
| lewo      |  środek   |     prawo |
| dane A    |  dane B   |    dane C |
Podgląd
Kolumna LKolumna CKolumna P
lewośrodekprawo
dane Adane Bdane C
Wyrównanie kolumn
Składnia separatora
:----   → wyrównanie do lewej
:----:  → wyśrodkowanie
----:   → wyrównanie do prawej
----    → domyślne (lewo)
Tip: Wewnątrz komórek można używać formatowania inline: **bold**, `kod`, linki. Nie można zagnieżdżać bloków kodu ani list.
➖Linia pozioma
Linia pozioma (<hr>) — separator sekcji. Trzy lub więcej gwiazdek, minusów lub podkreślników.
Markdown
Tekst powyżej

---

Tekst poniżej

***

Także linia pozioma

___
Podgląd

Tekst powyżej

Tekst poniżej

Także linia pozioma

Uwaga: --- bezpośrednio po tekście zostanie zinterpretowane jako nagłówek H2 (składnia Setext). Zawsze zostaw pustą linię przed ---.

Rozszerzenie GFM. Interaktywne checkboxy w README i issues GitHub. Składnia: - [ ] i - [x].
Markdown 
- [x] Zainstaluj zależności
- [x] Skonfiguruj .env
- [ ] Uruchom testy
- [ ] Wdróż na produkcję
Podgląd
  • ✓ Zainstaluj zależności
  • ✓ Skonfiguruj .env
  • Uruchom testy
  • Wdróż na produkcję
📎Przypisy (Footnotes)
Obsługiwane przez GitHub, Pandoc i wiele generatorów stron. Tworzą numerowane odnośniki na dole dokumentu.
Markdown
To jest zdanie z przypisem.[^1]
Kolejne zdanie z przypisem.[^nota]

[^1]: Treść pierwszego przypisu.
[^nota]: Przypis może zawierać
    wiele linii (wcięcie 4 spacje).
Podgląd

To jest zdanie z przypisem.[1]

Kolejne zdanie z przypisem.[2]

  1. Treść pierwszego przypisu.
  2. Przypis może zawierać wiele linii.
🚨Alerty GitHub
Specjalne bloki cytatów GFM renderowane jako kolorowe alerty. Obsługiwane na GitHub.com od 2023. Pięć typów.
Markdown 
> [!NOTE]
> Pomocna informacja dla użytkownika.

> [!TIP]
> Opcjonalna wskazówka lub porada.

> [!IMPORTANT]
> Kluczowa informacja dla użytkownika.

> [!WARNING]
> Krytyczna informacja wymagająca uwagi.

> [!CAUTION]
> Negatywne konsekwencje tej akcji.
Podgląd
ℹ NOTE

Pomocna informacja dla użytkownika.

💡 TIP

Opcjonalna wskazówka lub porada.

📌 IMPORTANT

Kluczowa informacja dla użytkownika.

⚠️ WARNING

Krytyczna informacja wymagająca uwagi.

🚨 CAUTION

Negatywne konsekwencje tej akcji.

Uwaga: Alerty działają tylko na GitHub.com i wybranych rendererach. W zwykłym pandoc lub mkdocs wyświetlą się jako zwykłe cytaty.
🏷️Odznaki (Badges)
Dynamiczne lub statyczne odznaki SVG z shields.io. Używane w README do prezentacji statusu, wersji, licencji, CI.
Markdown 
![wersja](https://img.shields.io/badge/version-1.0.0-blue)
![licencja](https://img.shields.io/badge/license-MIT-green)
![Python](https://img.shields.io/badge/Python-3.11-yellow?logo=python)
[![CI](https://github.com/.../badge.svg)](https://github.com/...)
Podgląd
version1.0.0 licenseMIT Python3.11 CIpassing
Struktura URL shields.io
Wzorzec
https://img.shields.io/badge/{label}-{message}-{color}

Kolory: brightgreen green yellow orange red blue purple
        success important critical informational inactive

Parametry:
  ?logo=python          → logo z simpleicons.org
  ?style=flat-square    → kształt (flat, flat-square, plastic, for-the-badge)
  ?labelColor=333       → kolor etykiety (hex)
🗺️Diagramy Mermaid
Mermaid.js renderuje diagramy z tekstu. Obsługiwany natywnie na GitHub, GitLab, Notion i wielu narzędziach dokumentacyjnych.
Flowchart
Markdown 
```mermaid
graph LR
  A[Start] --> B{Decyzja?}
  B -->|Tak| C[Akcja A]
  B -->|Nie| D[Akcja B]
  C --> E[Koniec]
  D --> E
```
Podgląd (schemat)
Start → Decyzja? →Tak→ Akcja A → Koniec
Renderowany diagram wymaga obsługi Mermaid.js w środowisku.
Typy diagramów
Obsługiwane typy
graph / flowchart   → schematy blokowe (LR, TD, RL, BT)
sequenceDiagram     → diagramy sekwencji
classDiagram        → diagramy klas UML
stateDiagram-v2     → maszyny stanów
erDiagram           → entity-relationship
gantt               → wykresy Gantta
pie                 → wykresy kołowe
gitGraph            → historia git
🧮Matematyka (LaTeX / KaTeX)
Wzory matematyczne w notacji LaTeX. GitHub obsługuje MathJax od 2022. Inline: $...$, blok: $$...$$.
Markdown 
Wzór inline: $E = mc^2$

Blok wzoru:
$$
\sum_{i=1}^{n} x_i = \frac{n(n+1)}{2}
$$
Podgląd

Wzór inline: E = mc²

∑ xᵢ = n(n+1)/2
Uwaga: Renderowanie LaTeX wymaga obsługi MathJax lub KaTeX. GitHub.com obsługuje natywnie. Lokalnie może wymagać pluginu (np. mkdocs-material).
🌐HTML inline
W większości rendererów Markdown dopuszcza surowy HTML. Przydatne dla funkcji niedostępnych w czystym MD: kolor, rozmiar obrazka, szczegóły/podsumowanie.
Sekcja rozwijana (details/summary)
Markdown 
<details>
<summary>Kliknij aby rozwinąć</summary>

Zawartość ukryta.
Można tu użyć **Markdown**.

</details>
Podgląd
Kliknij aby rozwinąć

Zawartość ukryta. Można tu użyć Markdown.

Inne użyteczne tagi HTML
Przydatne tagi
<kbd>Ctrl</kbd>+<kbd>C</kbd>   → skróty klawiszowe
<sub>dolny</sub>                → indeks dolny
<sup>górny</sup>                → indeks górny
<mark>zaznaczony</mark>         → podświetlenie (HTML5)
<br>                            → łamanie linii
<center>tekst</center>          → wyśrodkowanie (deprecated)
<img width="300" src="...">     → kontrola rozmiaru obrazka
<!-- komentarz -->              → niewidoczny komentarz
Markdown
Naciśnij <kbd>Ctrl</kbd>+<kbd>S</kbd> aby zapisać.
Podgląd

Naciśnij Ctrl+S aby zapisać.

🔒Escaping znaków specjalnych
Backslash \ przed znakiem specjalnym powoduje, że jest traktowany dosłownie. Alternatywnie — encje HTML.
Markdown
\*nie kursywa\*
\# nie nagłówek
\[nie link\](url)
\`nie kod\`

&amp;   → &
&lt;    → <
&gt;    → >
&copy;  → ©
&mdash; → —
Podgląd

*nie kursywa*

# nie nagłówek

[nie link](url)

`nie kod`

& < > © —

Znaki wymagające escapowania
\ ` * _ { } [ ] ( ) # + - . ! | > ~
📄Front Matter (YAML)
Blok YAML na początku pliku — metadane dokumentu. Obsługiwany przez Jekyll, Hugo, Gatsby, Obsidian, mkdocs i inne. Musi być pierwszym elementem pliku.
Markdown 
---
title: Tytuł dokumentu
date: 2024-01-15
author: Jan Kowalski
tags: [markdown, tutorial, docs]
draft: false
description: Krótki opis strony
---

# Treść dokumentu
Podgląd
FRONT MATTER (niewidoczny w podglądzie)
title: Tytuł dokumentu
date: 2024-01-15
tags: [markdown, tutorial, docs]

Treść dokumentu

Tip: Obsidian używa Front Matter do właściwości notatek. GitHub ignoruje Front Matter w podglądzie, ale używa go Jekyll i inne generatory. W samym pliku .md blok --- jest wyświetlany jako linia pozioma.
💡Dobre praktyki
Zasady pisania czytelnych i przenośnych plików Markdown.
Struktura dokumentu
✅ Zalecane
# Jeden H1 na dokument (tytuł)
## H2 dla głównych sekcji
### H3 dla podsekcji
#### H4 oszczędnie — tylko gdy naprawdę potrzebne

Puste linie między blokami (akapity, listy, bloki kodu).
Konsekwentny styl: albo - albo * dla list (nie mieszaj).
Linki referencyjne dla często powtarzających się URL.
Tekst alternatywny dla każdego obrazka.
Przenośność
Kompatybilność rendererów
CommonMark     → bazowy standard, wszędzie działa
GFM            → GitHub: tabele, task lists, alerty, strikethrough
Pandoc         → konwersja do PDF/DOCX/HTML, rich footnotes, math
Obsidian       → wikilinks [[strona]], callouts, embeds
mkdocs-material→ admonitions, tabs, annotations
Jekyll/Hugo    → shortcodes, front matter, templating
Częste błędy
❌ Unikaj
❌ Brak spacji po #  →  #Nagłówek (nie działa wszędzie)
❌ Mieszanie tabs i spacji w wcięciach list
❌ Brak pustej linii przed blokiem kodu lub listą
❌ Bezpośrednio po tekście --- (tworzy H2!)
❌ Tabele bez separatora  |---|  (linia 2 jest obowiązkowa)
❌ Obrazki bez tekstu alt  ![] zamiast ![opis]
❌ Bardzo długie linie (>120 znaków) — trudne w review
