🔤Nagłówki
Markdown obsługuje 6 poziomów nagłówków odpowiadających HTML <h1>–<h6>. Prefiks # — liczba znaków określa poziom. Zawsze zostaw spację po znakach #.
Markdown 
# Nagłówek H1
## Nagłówek H2
### Nagłówek H3
#### Nagłówek H4
##### Nagłówek H5
###### Nagłówek H6
Podgląd

Nagłówek H1

Nagłówek H2

Nagłówek H3

Nagłówek H4

Nagłówek H5
Nagłówek H6
Alternatywna składnia (Setext) dla H1 i H2
Markdown
Tytuł dokumentu
===============

Podtytuł
--------
Podgląd

Tytuł dokumentu

Podtytuł

Uwaga: H1 powinien wystąpić tylko raz w dokumencie — jest tytułem strony. Kolejne sekcje zaczynaj od H2.
✏️Formatowanie tekstu
Podstawowe style: pogrubienie, kursywa, przekreślenie, podkreślenie (HTML), indeksy oraz kod inline.
Markdown 
**pogrubiony**
*kursywa*
***pogrubiona kursywa***
~~przekreślony~~
`kod inline`
<u>podkreślony</u>
H~2~O  (GFM / Obsidian)
x^2^   (GFM / Obsidian)
Podgląd

pogrubiony

kursywa

pogrubiona kursywa

przekreślony

kod inline

podkreślony

H2O

x2

Akapity i łamanie linii
Markdown
Pierwszy akapit tekstu.

Drugi akapit — pusta linia tworzy akapit.

Pierwsza linia··
Druga linia (2 spacje = <br>)
Podgląd

Pierwszy akapit tekstu.

Drugi akapit — pusta linia tworzy akapit.

Pierwsza linia
Druga linia (2 spacje = <br>)

Tip: Możesz też używać _kursywa_ i __pogrubienie__ z podkreślnikami — ale gwiazdki są bardziej universalne i rzadziej powodują problemy w środku słów.
📋Listy
Nieuporządkowane (punktorowe), uporządkowane (numerowane), zagnieżdżone i mieszane. Wcięcie 2 lub 4 spacje tworzy poziom zagnieżdżenia.
Lista nieuporządkowana
Markdown
- Element A
- Element B
  - Zagnieżdżony 1
  - Zagnieżdżony 2
    - Głębszy poziom
- Element C
Podgląd
  • Element A
  • Element B
    • Zagnieżdżony 1
    • Zagnieżdżony 2
      • Głębszy poziom
  • Element C
Lista uporządkowana
Markdown
1. Krok pierwszy
2. Krok drugi
   1. Podkrok 2a
   2. Podkrok 2b
3. Krok trzeci
Podgląd
  1. Krok pierwszy
  2. Krok drugi
    1. Podkrok 2a
    2. Podkrok 2b
  3. Krok trzeci
Uwaga: Liczby w liście numerowanej nie muszą być kolejne — Markdown sam je ponumeruje. Możesz pisać 1. 1. 1. i efekt będzie taki sam.

tekst linku

https://example.com

Obrazy

> Może obejmować wiele linii.

💻Kod
Kod inline (backtick) i bloki kodu (triple backtick z nazwą języka). Bloki kodu obsługują syntax highlighting na GitHub, GitLab i w większości rendererów.
Kod inline
Markdown
Użyj `npm install` aby zainstalować.
Zmienna `const x = 42` jest stała.
Podgląd

Użyj npm install aby zainstalować.

Zmienna const x = 42 jest stała.

Blok kodu z podświetlaniem składni
Markdown
```python
def hello(name: str) -> str:
    return f"Hello, {name}!"

print(hello("World"))
```
Podgląd
def hello(name: str) -> str:
    return f"Hello, {name}!"

print(hello("World"))

Blok diff (zmiany)
Markdown

📊Tabele
Tabele GFM (GitHub Flavored Markdown). Wyrównanie kolumn kontroluje się dwukropkiem w linii separatora.
Markdown 
| Kolumna L | Kolumna C | Kolumna P |
| :-------- | :-------: | --------: |
| lewo      |  środek   |     prawo |
| dane A    |  dane B   |    dane C |
Podgląd
Kolumna LKolumna CKolumna P
lewośrodekprawo
dane Adane Bdane C
Wyrównanie kolumn
Składnia separatora
:----   → wyrównanie do lewej
:----:  → wyśrodkowanie
----:   → wyrównanie do prawej
----    → domyślne (lewo)
Tip: Wewnątrz komórek można używać formatowania inline: **bold**, `kod`, linki. Nie można zagnieżdżać bloków kodu ani list.

Linia pozioma (<hr>) — separator sekcji. Trzy lub więcej gwiazdek, minusów lub podkreślników.
Markdown
Tekst powyżej

---

Tekst poniżej

***

Także linia pozioma

___
Podgląd

Tekst powyżej

Tekst poniżej

Także linia pozioma

🚨Alerty GitHub
Specjalne bloki cytatów GFM renderowane jako kolorowe alerty. Obsługiwane na GitHub.com od 2023. Pięć typów.
Markdown 
> [!NOTE]
> Pomocna informacja dla użytkownika.

> [!TIP]
> Opcjonalna wskazówka lub porada.

> [!IMPORTANT]
> Kluczowa informacja dla użytkownika.

> [!WARNING]
> Krytyczna informacja wymagająca uwagi.

> [!CAUTION]
> Negatywne konsekwencje tej akcji.
Podgląd
ℹ NOTE

Pomocna informacja dla użytkownika.

💡 TIP

Opcjonalna wskazówka lub porada.

📌 IMPORTANT

Kluczowa informacja dla użytkownika.

⚠️ WARNING

Krytyczna informacja wymagająca uwagi.

🚨 CAUTION

Negatywne konsekwencje tej akcji.

Uwaga: Alerty działają tylko na GitHub.com i wybranych rendererach. W zwykłym pandoc lub mkdocs wyświetlą się jako zwykłe cytaty.

Markdown

Blok wzoru:
$$
\sum_{i=1}^{n} x_i = \frac{n(n+1)}{2}
$$

Etap 01
Discovery

Wstępne rozpoznanie — zrozumienie kontekstu biznesowego, zebranie założeń i zidentyfikowanie obszarów niewiedzy. AI pomaga szybko ustrukturyzować chaos na początku projektu.

📄 Brief · 🗺️ Mapa założeń · ❓ Lista pytań badawczych · 📋 Scope projektu
PromptAnaliza briefu
Przeanalizuj poniższy brief projektowy. Zidentyfikuj: 1. Główny problem biznesowy (1–2 zdania) 2. Założenia, które są faktem vs. założenia, które wymagają weryfikacji 3. Grupę docelową — co wiemy, czego nie wiemy 4. Ryzyka i obszary niejasności 5. Pytania, które koniecznie zadałby doświadczony badacz UX przed startem projektu Brief: [WKLEJ BRIEF]
PromptMapa założeń
Na podstawie poniższych notatek ze spotkania kickoff stwórz mapę założeń projektu w formacie tabeli Markdown. Kolumny: Założenie | Kategoria (biznesowe / użytkownik / techniczne) | Pewność (wysoka / średnia / niska) | Sposób weryfikacji. Notatki: [WKLEJ NOTATKI]
PromptScope projektu
Pomóż mi sformułować zakres projektu (scope). Na podstawie opisu stwórz: - In scope: lista rzeczy, które robimy - Out of scope: lista rzeczy, które świadomie pomijamy - Otwarte decyzje: rzeczy do ustalenia z klientem Opis projektu: [OPIS]
Etap 02
Research

Badania użytkowników, analiza konkurencji, desk research. AI nie zastępuje rozmów z użytkownikami, ale radykalnie przyspiesza przygotowanie, analizę i syntezę.

🎤 Scenariusz wywiadu · 📊 Synteza badań · 👤 Persony · 🗺️ Customer Journey Map
PromptScenariusz wywiadu
Stwórz scenariusz wywiadu pogłębionego (IDI) z użytkownikiem. Cel badania: [CEL]. Grupa docelowa: [OPIS GRUPY]. Scenariusz powinien zawierać: - Rozgrzewkę (3–4 pytania otwierające) - Sekcję główną (8–10 pytań pogłębiających, bez pytań sugestywnych) - Pytania o obejścia (workarounds) i frustracje - Zamknięcie Każde pytanie opatrz wskazówką dla moderatora w nawiasie.
PromptSynteza wywiadów
Poniżej masz transkrypcje [N] wywiadów z użytkownikami. Przeprowadź syntezę: 1. Zidentyfikuj powtarzające się tematy (minimum 3 wystąpienia) 2. Wypisz kluczowe cytaty ilustrujące każdy temat 3. Wskaż napięcia i sprzeczności między respondentami 4. Sformułuj 3–5 kluczowych insightów w formacie: "Użytkownicy [zachowanie], ponieważ [przyczyna], co oznacza, że [implikacja dla designu]" Transkrypcje: [WKLEJ TRANSKRYPCJE]
PromptAnaliza konkurencji

Jesteś doświadczonym Senior Product Designerem z 10+ latami praktyki w produktach cyfrowych. Twoja rola to bycie wymagającym, konstruktywnym partnerem — nie asystentem, który potakuje.

Kiedy oceniasz pomysły lub projekty:

Nie mów "świetny pomysł" jeśli masz wątpliwości. Mów co widzisz, czego brakuje i co zrobiłbyś inaczej. Twoja wartość to uczciwa, oparta na doświadczeniu perspektywa — nie komplementy.

⚠️ Kiedy nie używać AI
Szczera lista sytuacji, w których AI szkodzi zamiast pomagać. Nie dlatego że AI jest złe — ale dlatego że każde narzędzie ma swój zakres zastosowania.
🚫
Nigdy nie zastąpisz rozmowy z użytkownikiem
AI może symulować użytkownika, ale nie zastąpi prawdziwej obserwacji. Modele językowe odpowiadają jak "przeciętny człowiek z internetu" — nie jak Twój konkretny użytkownik z jego kontekstem, emocjami i otoczeniem. Syntetyczny research to brak researchu z etykietą researchu.
Zasada: AI może przygotować pytania i analizować notatki. Rozmowę przeprowadza człowiek.
🚫
Decyzje z konsekwencjami prawnymi lub etycznymi
AI nie zna aktualnego prawa Twojej jurysdykcji, nie ma dostępu do najnowszych regulacji i nie ponosi odpowiedzialności za błąd. RODO, dyrektywy dostępności, prawo pracy, umowy — to wymaga prawnika, nie modelu językowego.
Zasada: AI może pomóc sformułować pytanie do prawnika. Odpowiedź prawna pochodzi od prawnika.
⚠️
Fakty o konkretnych danych Twojej firmy
Model nie zna Twojego NPS, retention rate, churn, wyników testów A/B ani road mapy. Gdy pytasz o "typowe" metryki branżowe — dostajesz uśrednione dane z internetu, które mogą nie mieć związku z Twoim kontekstem.
Zasada: Dostarczaj dane z systemu. Nie pytaj AI o dane — pytaj o interpretację danych które sam podajesz.
⚠️
Walidacja pomysłów przez "AI jako użytkownik"
Pytanie AI "czy użytkownicy polubią tę funkcję?" daje fałszywe poczucie walidacji. AI odpowiada na podstawie tego co jest popularne w tekstach z internetu, nie na podstawie Twojego segmentu. To confirmation bias w skali.
Zasada: AI może wskazać potencjalne problemy z pomysłem. Nie może potwierdzić że pomysł zadziała.
⚠️
Ocena jakości własnego designu bez kontekstu
AI oceni design na podstawie ogólnych zasad UX z internetu. Nie zna: celów biznesowych, grupy docelowej, ograniczeń technicznych, historii produktu ani poprzednich decyzji. Taka ocena jest tak dobra jak opinia przypadkowej osoby bez briefu.
Zasada: Podawaj maksymalny kontekst. Im mniej kontekstu — tym mniej warta ocena.
💡
Gdy AI odpowiada pewnie, a Ty tego nie weryfikujesz
Modele językowe są zaprojektowane żeby brzmieć pewnie — nawet gdy się mylą. Halucynacje dotyczące statystyk, nazw firm, dat, cytatów i specyfiki narzędzi są powszechne i nie są poprzedzone ostrzeżeniem. Pewność tonu ≠ poprawność faktu.
Zasada: Każdy konkretny fakt, statystykę lub cytat z odpowiedzi AI weryfikuj w źródle pierwotnym.
💡
Kreatywność jako substytut rozumienia problemu
Generowanie 20 pomysłów w 30 sekund jest kuszące, ale pomysły bez głębokiego rozumienia problemu są bezwartościowe. AI może wyprodukować dużo słów — ale nie zastąpi czasu spędzonego na rozumieniu dlaczego coś jest problemem.
Zasada: Używaj AI do ideacji po — nie zamiast — etapu rozumienia problemu.
💡
Automatyzacja decyzji które powinny być ludzkie
Priorytetyzacja, wybór kierunku produktu, decyzje o tym kogo słuchać i kogo nie — to wymaga ludzkiej odpowiedzialności i kontekstu organizacyjnego. AI może pomóc ustrukturyzować myślenie, ale nie może (i nie powinno) decydować za Ciebie.
Zasada: AI dostarcza perspektyw i struktury. Decyzja i odpowiedzialność pozostaje po stronie człowieka.
Dobry test: Przed użyciem AI zapytaj siebie — "czy dostarczam wystarczający kontekst żeby AI mogło dać mi użyteczną odpowiedź, czy oczekuję że AI zastąpi moje myślenie?" Pierwsze jest narzędziem. Drugie jest ryzykiem.
